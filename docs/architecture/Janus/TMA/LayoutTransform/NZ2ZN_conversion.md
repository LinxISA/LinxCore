# NZ→ZN 分型转置规格

## 1. 概述

Cube 消费矩阵数据时，左矩阵需要 NZ 格式，右矩阵需要 ZN 格式。NZ→ZN 转置级联在 ND'→NZ 转换（TPU 输出）与 Cube 消费之间。Cube 输出始终为 NZ，无需反向转置。

```
数据路径（Cube 输入侧）：
  TR(ND') → TPU(ND'→NZ) → 转置逻辑(NZ→ZN) → Cube

数据路径（Cube 输出侧）：
  Cube → NZ 数据 → TPU(NZ→ND') → TR(ND')
```

## 2. ZN 格式定义

| 属性 | NZ | ZN |
|------|----|----|
| Fractal 内 | 行优先 | 列优先 |
| Fractal 间 | 列优先 | 行优先 |
| 逻辑形状（转置后） | 16 × 32B | 16 × 32B |
| 形状（转置前） | 16 × 32B | 2B: 16×32B<br>1B: 32×16B<br>0.5B: 64×8B |

ZN 是 NZ 的元素级转置：NZ 中位于 (row, col) 的元素，在 ZN 中位于 (col, row)。

## 3. 转置区域：64 × 32B

转置改变矩阵形状（非方阵时）。为保证输入区域与输出区域字节数相同且形状统一，选取 **64 行 × 32B = 2048B** 作为转置操作的固定区域。不同元素类型下，该区域内的分型排布方式不同，但总尺寸恒为 64×32B。转置后统一为 R4C1 的 4 个 16×32B ZN Fractal。

### 3.1 为何无法逐 NZ Fractal 独立转置

| 数据类型 | NZ Fractal 元素形状 | 转置块形状 | 一个 NZ Fractal 贡献给几个转置块 |
|---------|-------------------|-----------|-------------------------------|
| 16-bit | 16×16（方阵） | 16×16 | 1（可独立） |
| 8-bit | 16×32 | 32×16 | 2（左半+右半分属不同块） |
| 4-bit | 16×64 | 64×16 | 4（每 16 列分属一个块） |

8-bit 和 4-bit 下，一个 NZ Fractal 的列被拆分到多个转置块中，必须等所有相关 NZ Fractal 到齐后才能开始转置。

## 4. 数据流时序

TPU 带宽 2KB/cycle（= 4 Fractal/cycle = 64×32B/cycle），恰好等于一个转置区域的大小。

### 4.1 对齐情况：1 cycle 处理

当 TPU 从 TR 读出的 2KB 数据恰好构成一个完整的 64×32B 转置区域时：

```
Cycle 0: TPU 读出 2KB（4 个 NZ Fractal）→ 转置逻辑收到完整 64×32B 区域
Cycle 1: 转置逻辑处理 → 输出 2KB（4 个 ZN Fractal）给 Cube
```

上游带宽（2KB/cy）与转置处理带宽（64×32B/cy）完全匹配，无需缓冲。

### 4.2 不对齐情况：需缓冲

当 TPU 读出的 2KB 数据跨越两个不同的 64×32B 转置区域（各取一半）时：

```
Cycle 0: TPU 读出 2KB = 区域 A 的后半 (1KB) + 区域 B 的前半 (1KB)
         → 区域 A 前半已缓冲，凑齐 → 转置区域 A
         → 区域 B 前半缓冲，等待
Cycle 1: TPU 读出 2KB = 区域 B 的后半 (1KB) + 区域 C 的前半 (1KB)
         → 区域 B 凑齐 → 转置区域 B
         → 区域 C 前半缓冲，等待
```

稳态下仍为 1 个转置区域/cycle（流水线），但需要 2KB 的缓冲空间，且首个区域有 +1 cycle 延迟。
