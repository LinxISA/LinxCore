# LinxCore Microarchitecture Conformance Audit

Date: 2026-03-11

Scope:

- Canonical contract source: `rtl/LinxCore/docs/architecture/*`
- Static implementation review across backend/decode/dispatch/issue/trace
- Targeted runtime evidence for block/BID and standalone CoreMark divergence

## Executive verdict

Overall conformance: `does not conform`

Highest-risk reasons:

1. `LC-MA-BLK-001` is runtime-broken in the current standalone no-stub lane. The active block domain does not advance at the `_start -> _start_c` handoff, younger wrong-path rows survive `BSTOP`, and the DUT diverges from QEMU at `seq=8`.
2. `LC-MA-HAZ-001` is not implemented as specified in the live IQ/issue path. Required `inflight`, speculative-ready, `ld_gen_vec`, and `miss_pending` state are absent, and IQ entries deallocate on pick/issue instead of at `I2`.
3. `LC-IF-TRACE-001` and the module/stage ownership contract are partially undermined by a custom trace-export path that still reconstructs synthetic boundary/trace behavior in the monolithic backend wrapper instead of relying only on the dedicated owner stages the spec names.

The core does have conformant pieces, especially around canonical docs inside the submodule, stage-token connectivity checks, and the `D2` immediate-only producer path for `SETRET` / `C.SETRET`. But the existing implementation does not meet the full `v0.4` microarchitecture contract.

## Confirmed non-conformances

### 1. `bug` - `LC-MA-BLK-001`: `BSTART` does not carry the new BID in the live backend path

Type: static contradiction plus runtime-confirmed failure

Contract:

- [microarchitecture.md](/Users/zhoubot/linx-isa/rtl/LinxCore/docs/architecture/microarchitecture.md#L460) says BID is generated by `BROB`, flush is BID-based, and [microarchitecture.md](/Users/zhoubot/linx-isa/rtl/LinxCore/docs/architecture/microarchitecture.md#L493) says `BSTART` carries the new BID of the new block.
- [interfaces.md](/Users/zhoubot/linx-isa/rtl/LinxCore/docs/architecture/interfaces.md#L79) requires `BID` to remain visible through enqueue, issue, completion, and flush rollback.

Implementation evidence:

- [trace_export_core.py](/Users/zhoubot/linx-isa/rtl/LinxCore/src/bcc/backend/modules/trace_export_core.py#L2244) computes `slot_take_new_bid`, but [trace_export_core.py](/Users/zhoubot/linx-isa/rtl/LinxCore/src/bcc/backend/modules/trace_export_core.py#L2245) immediately documents and implements the opposite behavior: the `BSTART` slot stays on the live block identity and only younger same-packet slots inherit the new BID.

Runtime evidence:

- Standalone no-stub failfast aborts with `xcheck mismatch at seq=8 field=pc qemu=0x124AA dut=0x124DE`.
- In the same run, backend focus shows `brob_alloc_fire` and `assign_block_bid` advancing (`asn=0x11`, `0x22`, `0x33`, `0x44`), while `active_block_bid` and `flush_bid` remain `0x0`.
- Wrong-path rows `0x124A6/0x124A8` survive after `C.BSTOP @ 0x124A4`, and a later wrong-path row at `0x124DE` retires instead of flushing.

Conclusion:

The current implementation violates the normative `BSTART=new BID` contract, and that violation is the most likely root cause of the confirmed block/flush divergence.

### 2. `bug` - `LC-MA-BLK-001` and `LC-MA-HAZ-001`: `BSTOP` redirect/flush does not kill younger wrong-path work in standalone no-stub mode

Type: runtime-confirmed

Contract:

- [microarchitecture.md](/Users/zhoubot/linx-isa/rtl/LinxCore/docs/architecture/microarchitecture.md#L424) makes boundary semantics authoritative for redirect and retirement.
- [microarchitecture.md](/Users/zhoubot/linx-isa/rtl/LinxCore/docs/architecture/microarchitecture.md#L468) requires `keep bid <= flush_bid, kill bid > flush_bid`.
- [microarchitecture.md](/Users/zhoubot/linx-isa/rtl/LinxCore/docs/architecture/microarchitecture.md#L326) forbids committing younger wrong-path state.

Runtime evidence:

- In the failfast run, after the `0x1249C -> 0x124A4` call/BSTOP window, the ROB head advances into `0x124A6` and then later `0x124DE` instead of flushing cleanly to `_start_c @ 0x124AA`.
- The debug dump shows `rob_head_pc=0x124A6` at cycle `5083 post`, then `rob_head_pc=0x124DE` by cycle `5089 post`, while architectural control is already supposed to be at `0x124AA`.

Relevant implementation area:

- [commit_slot_step.py](/Users/zhoubot/linx-isa/rtl/LinxCore/src/bcc/backend/modules/commit_slot_step.py)
- [trace_export_core.py](/Users/zhoubot/linx-isa/rtl/LinxCore/src/bcc/backend/modules/trace_export_core.py)

Conclusion:

The current redirect/flush path is not preserving the correct block domain through the call-to-`_start_c` handoff.

### 3. `bug` - `LC-MA-HAZ-001`: live IQ path deallocates on issue pick/fire instead of at `I2`, and required `inflight` state is missing

Type: static-only

Contract:

- [microarchitecture.md](/Users/zhoubot/linx-isa/rtl/LinxCore/docs/architecture/microarchitecture.md#L328) requires entries to remain valid while `inflight`.
- [microarchitecture.md](/Users/zhoubot/linx-isa/rtl/LinxCore/docs/architecture/microarchitecture.md#L345) makes `I2` the deallocation point.
- [pipeline-stage-catalog.md](/Users/zhoubot/linx-isa/rtl/LinxCore/docs/architecture/pipeline-stage-catalog.md#L97) says IQ owns oldest-first pick and `inflight` residency.

Implementation evidence:

- [iq_bank.py](/Users/zhoubot/linx-isa/rtl/LinxCore/src/bcc/backend/modules/iq_bank.py#L148) clears an entry on flush or `issue_fire`.
- [iq_bank.py](/Users/zhoubot/linx-isa/rtl/LinxCore/src/bcc/backend/modules/iq_bank.py#L150) ties clear directly to `issue_fire[slot] & issue_idx[slot] == i`.
- The file has no `inflight` state at all; a repo search in `src/bcc/backend` and `src/bcc/lsu` found no IQ `inflight` implementation except unrelated `scb_inflight`.
- [issue.py](/Users/zhoubot/linx-isa/rtl/LinxCore/src/bcc/backend/issue.py#L180) builds `*_can_issue` directly from valid+ready, with no `inflight` exclusion state and no later `I2` ownership before deallocation.

Conclusion:

This is a direct mismatch to the documented IQ residency model, not just an implementation simplification.

### 4. `bug` - `LC-MA-HAZ-001`: same-cycle wakeup can feed pick in the active backend path

Type: static-only

Contract:

- [microarchitecture.md](/Users/zhoubot/linx-isa/rtl/LinxCore/docs/architecture/microarchitecture.md#L356) explicitly says wakeup at cycle `N` must only be visible to pick at cycle `N+1`.

Implementation evidence:

- [trace_export_core.py](/Users/zhoubot/linx-isa/rtl/LinxCore/src/bcc/backend/modules/trace_export_core.py#L2385) builds `wb_set_mask` from same-cycle issue writebacks and same-cycle `disp_resolved_d2` producers.
- [trace_export_core.py](/Users/zhoubot/linx-isa/rtl/LinxCore/src/bcc/backend/modules/trace_export_core.py#L2393) assigns that mask straight into `ren_wb_set_mask`.
- The current IQ banks consume a flat `ready_mask` and compute pick eligibility combinationally from it in [iq_bank.py](/Users/zhoubot/linx-isa/rtl/LinxCore/src/bcc/backend/modules/iq_bank.py#L96).

Conclusion:

The live wiring strongly suggests a same-cycle wake-to-pick path, which violates the architectural timing rule.

### 5. `bug` - `LC-MA-HAZ-001`: required speculative-ready, `ld_gen_vec`, and `miss_pending` behavior is absent from the IQ/issue implementation

Type: static-only

Contract:

- [microarchitecture.md](/Users/zhoubot/linx-isa/rtl/LinxCore/docs/architecture/microarchitecture.md#L349) requires speculative readiness in IQ-entry state via `is_spec`.
- [microarchitecture.md](/Users/zhoubot/linx-isa/rtl/LinxCore/docs/architecture/microarchitecture.md#L396) requires load speculative wakeup, `ld_gen_vec`, and `miss_pending` suppression.

Implementation evidence:

- A repo search over `src/bcc/backend` and `src/bcc/lsu` found no live implementation references for `is_spec`, `ld_gen_vec`, or `miss_pending` in the IQ/issue path.
- [issue.py](/Users/zhoubot/linx-isa/rtl/LinxCore/src/bcc/backend/issue.py#L195) reduces LSU issue eligibility to ordinary ready bits and load/store classification only.
- [iq_bank.py](/Users/zhoubot/linx-isa/rtl/LinxCore/src/bcc/backend/modules/iq_bank.py#L97) models readiness as simple `mask_bit` checks over a global ready mask.

Conclusion:

The live issue machinery does not implement the documented speculative-load hazard model.

### 6. `spec drift` - `LC-MA-STAGE-001` and `LC-MA-PIPE-001`: canonical backend ownership is still concentrated in the monolithic trace-export core

Type: static-only

Contract:

- [microarchitecture.md](/Users/zhoubot/linx-isa/rtl/LinxCore/docs/architecture/microarchitecture.md#L50) makes module decomposition part of the architectural contract.
- [module-catalog.md](/Users/zhoubot/linx-isa/rtl/LinxCore/docs/architecture/module-catalog.md#L11) says architecturally visible stage and ownership logic must live in dedicated modules and top shells should converge toward connection-only composition.
- [pipeline-stage-catalog.md](/Users/zhoubot/linx-isa/rtl/LinxCore/docs/architecture/pipeline-stage-catalog.md#L13) says visible stages must remain backed by real owner modules.

Implementation evidence:

- [backend.py](/Users/zhoubot/linx-isa/rtl/LinxCore/src/bcc/backend/backend.py#L11) says canonical backend composition lives in the trace-export path and then delegates the backend to [trace_export_core.py](/Users/zhoubot/linx-isa/rtl/LinxCore/src/bcc/backend/modules/trace_export_core.py#L3144).
- [trace_export_core.py](/Users/zhoubot/linx-isa/rtl/LinxCore/src/bcc/backend/modules/trace_export_core.py#L638) still acts as a large monolithic owner for dispatch, rename wiring, IQ banks, commit shaping, CTU interaction, trace export, and debug plumbing.

Conclusion:

The canonical docs and the actual backend decomposition are still materially different. This is not the highest-severity functional bug, but it is a real conformance failure against the documented ownership model.

### 8. `spec drift` - `LC-IF-TRACE-001`: trace-export logic still synthesizes boundary/trace rows inside the monolithic wrapper

Type: static-only

Conclusion:

The current trace-export path still reconstructs architectural events in the parent wrapper instead of fully relying on the dedicated stage-owner path the contract describes.

### 3. IQ flush compare itself uses the correct directional BID kill rule

Contract:

- [microarchitecture.md](/Users/zhoubot/linx-isa/rtl/LinxCore/docs/architecture/microarchitecture.md#L468) requires `keep bid <= flush_bid, kill bid > flush_bid`.

Implementation evidence:

- [iq_bank.py](/Users/zhoubot/linx-isa/rtl/LinxCore/src/bcc/backend/modules/iq_bank.py#L149) clears an entry when `flush_bid < block_bid[i]`, which is directionally consistent with `kill bid > flush_bid`.

Verdict:

This isolated compare is correct. The observed standalone failure comes from the upstream block-domain state never advancing correctly, not from this particular IQ kill comparator.

## Bottom line

The current LinxCore tree partially matches the written `v0.4` contract in decode, stage-token naming, and some queue-local mechanics, but it does not conform overall. The decisive break is block/BID/flush behavior in the standalone no-stub lane, reinforced by a second cluster of static non-conformances in the IQ hazard model and by stage/trace ownership drift in the canonical backend implementation.
